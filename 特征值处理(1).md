### 1. Flink 计算后推送到 Redis 还是 Kafka？
**结论：推荐推送到 Redis，或者“先 Kafka 后 Redis”。**

*   **为什么首选 Redis？**

*   **什么时候用 Kafka？**

### 一、两个模型与“报警”的关系
#### 1. 故障概率模型：需要**全量数据**（不管是否报警）
- **核心逻辑**：它的作用是**“早期预警”**，即在设备还没触发报警之前，就识别出“亚健康”状态。
- **如果只收报警时的特征**：模型只能学到“报警时是什么样”，无法学到“正常→亚健康”的过渡模式，就会变成一个“事后诸葛亮”（报警了才报故障），完全失去预测价值。
- **正确做法**：Flink 要把设备**所有运行状态**的特征都推过来（包括正常运行、轻载、重载、待机等）。模型需要从这些数据里学会区分“正常重载（干活）”和“异常过载（故障）”。

#### 2. 剩余寿命（RUL）模型：需要**全生命周期数据**（从健康到故障的连续数据）
- **核心逻辑**：它的作用是**“趋势预测”**，即根据设备的退化过程，预测还能用多久。
- **如果只收报警时的特征**：模型只能看到“快坏了”的状态，看不到“健康→亚健康→故障”的退化曲线，根本无法拟合剩余寿命。
- **正确做法**：Flink 要持续收集设备从新机床到快报废的全阶段特征（尤其是健康阶段的基线数据，以及亚健康阶段的退化信号）。

### 🛠️ Flink 实现建议（核心代码逻辑）

在 Flink 中，建议使用 **`KeyedProcessFunction`** 来处理这个逻辑，因为我们需要按键（`BaseID`）维护状态。

```java
public class RulFeatureProcessFunction extends KeyedProcessFunction<String, ChangeRecord, RulFeatureRow> {

    // 1. 定义累加状态
    private ValueState<Double> cumulativeRuntimeState;
    private ValueState<Double> cumulativePartsState;
    
    // 2. 定义滑动窗口状态 (使用 ListState 存储最近 10 条记录)
    private ListState<Double> recentLoadsState;
    private ListState<Double> recentTimesState;

    @Override
    public void processElement(ChangeRecord record, Context ctx, Collector<RulFeatureRow> out) throws Exception {
        
        // --- A. 基础解析 ---
        double duration = record.getEnd() - record.getStart();
        double load = parseXml(record.getData(), "主轴负载");
        double cutTime = parseXml(record.getData(), "切削时间");
        
        // --- B. 累加计算 ---
        double cumRuntime = cumulativeRuntimeState.value() == null ? 0 : cumulativeRuntimeState.value();
        cumRuntime += duration;
        cumulativeRuntimeState.update(cumRuntime);
        
        // --- C. 滑动窗口计算 (以 avg_spindle_load_10 为例) ---
        // 1. 将新数据加入 List
        recentLoadsState.add(load);
        
        // 2. 如果超过 10 个，移除最早的 (ListState 不支持直接删头，建议用 ValueState<List<Double>> 或者自定义队列)
        List<Double> loads = new ArrayList<>();
        for(Double val : recentLoadsState.get()) loads.add(val);
        if (loads.size() > 10) loads.remove(0); 
        
        // 3. 计算均值
        double avgLoad10 = loads.stream().mapToDouble(Double::doubleValue).average().orElse(0.0);
        
        // --- D. 组装输出 ---
        RulFeatureRow feature = new RulFeatureRow(
            record.getBaseId(),
            duration, 
            cumRuntime, 
            avgLoad10, 
            // ... 其他特征
        );
        
        out.collect(feature);
    }
}
```

3.  **特征精简建议**：
    *   `is_running`, `is_standby` 等状态特征，如果当前是“离线”状态，通常不进行寿命预测。可以在 Flink 端加一个 Filter，**只在设备“运行”状态下推送特征**，节省 Java 计算资源。

**总结**：Flink 的核心工作是**维护 `BaseID` 维度的累加器和 10 条记录的滑动队列**。这套逻辑非常轻量，完全不会阻塞实时流。
